### 1. Tuning primitives write results to session.state BEFORE the judge gate; a failed-but-skipped step poisons downstream steps

`atomize/epr_auto/primitives/tune.py:853` — **CONFIRMED** · risk · dimension: error-paths

**Defect.** The step-layer judge gate lives in the CALLER: _run_primitive (steps.py:83-93) receives (result, judge_reports) and only THEN raises StepFailure on a failed hard judge. So any primitive that mutates session.state and returns a failing judge has already committed the bad result. tune.pi_calibration writes `session.state['pi_calibration'] = result` unconditionally (tune.py:766, 845, 853) even when it railed (rails='high' sets pi=x[-1], pi2 a root — both non-None); tune.auto_phase writes session.state['auto_phase'] (tune.py:187) before returning phase_coherence; tune.echo_window writes session.state['echo_window'] (tune.py:346) before returning echo_in_trace. By contrast field.edfs and tune.rep_rate DELIBERATELY guard their state writes behind success (field.py only sets session.state['field'] on the echo-found path; tune.rep_rate stores only on the 'covered'/'flat' branches, not on fit-failed/saturated) — the three tuning primitives above do not, so the invariant 'session.state holds only accepted tuning' is violated. When such a step FAILS its hard judge but the failure is not fatal to the run (on_fail: skip), _do_step leaves the rejected value in session.state, and a later exp.* _apply_cal (steps.py:373-404 -> tune.apply_calibration) reads session.state['pi_calibration'] and patches the experiment preset's pulse amplitudes with it, driving the rejected calibration to hardware with no error. apply_calibration only rejects None pi/pi2, not a railed one. The rail-triggered coarse fallback masks this in the default on_fail:abort flow, so the hazard is specific to on_fail:skip on a tuning step (a documented, first-class policy) or a fallback that is unavailable.

**Scenario.** Protocol: `tune.pi_calibration: {mode: amplitude, on_fail: skip}` followed by `exp.t2` with inferred apply_cal, and no earlier tune.power_for_length (so the rail fallback is unavailable). The resonator is underpowered, pi rails high: amplitude_rails FAILs, _run_primitive raises StepFailure, retries exhaust, on_fail:skip records 'failed-skipped' and continues — but session.state['pi_calibration'] already holds pi≈99% (the railed sweep max). exp.t2's _apply_cal infers {P2:pi2,P3:pi} and sets the Hahn pulses to the railed 99% (or raises >100% on a length mismatch), instead of falling back to the preset's trusted stored amplitudes as 'skip' implies. Silent wrong-flip-angle experiment reaching hardware.

**Suggested fix.** Make the state write conditional on the step being accepted — either move the session.state assignment out of the primitives into _do_step (which already writes session.state[step.name]=result only on status=='ok'), or have the primitives stage the result and let the runner commit it only after the judge gate passes. Mirror the guard field.edfs/rep_rate already use.

### 2. Same-day re-run reuses the run directory: manifest.json and NNN_tag.csv silently overwritten

`atomize/epr_auto/session.py:86` — **CONFIRMED** · bug · dimension: state-flow

**Defect.** session.run_dir defaults to ~/epr_data/epr_auto_{date}_{sample} (session.py:79-87) — deterministic per calendar day and sample, with no run-unique component; the manifest path is always run_dir/manifest.json (runner.py:75) and the protocol snapshot always protocol_<name> (runner.py:77), both plain overwrites; and _save_counter restarts at 0 in every new CLI process (session.py:31, 94-97), so the second run's files reuse 001_..., 002_... names and the worker's save (answered via 'FL<path>', executor.py:126) overwrites them. The documented crash-safety promise of the manifest ('a crash still leaves a valid record', runner.py:18-21) is defeated by the very next invocation. Files from run 1 that run 2 does not reach remain mixed in the directory with no marker of which manifest they belong to.

**Scenario.** Overnight protocol aborts at step 7 (or the operator dry-runs, then goes live, then re-runs after fixing a preset — the routine loop). Second `epr-auto run` the same day with the same sample: run 1's manifest.json and 001_edfs.csv..006_*.csv are silently replaced; the surviving 007+ files from run 1 sit beside run 2's manifest, which does not describe them. The record of what the aborted run did to the hardware (vane moves, field) is gone.

**Suggested fix.** Add a run-unique component (HHMMSS timestamp or _run2 suffix probe) to the default run_dir, or at least refuse/rename when manifest.json already exists in the target directory.

### 3. power_for_length discards the internal nutation's judges — vane iterates on noise with no echo gate

`atomize/epr_auto/primitives/tune.py:908` — **CONFIRMED** · risk · dimension: state-flow

**Defect.** In the power_for_length loop, `result, cal_judges = pi_calibration(...)` (tune.py:908) throws cal_judges away: the internal echo_snr and fit_quality verdicts are never gated, never logged, and never reach the manifest (session.last_judges is only set by _run_primitive at the step layer). The only hard gate is pi_length_target at the END (blessed decision, ROADMAP), but nothing stops up to max_iter-1 REAL vane moves computed from noise: with no echo, the fit fails and the fallback argmin-of-smoothed-noise (tune.py:813) yields a random 'measured' pi, and dB += 20*log10(target/measured) (tune.py:932) can command the vane toward 0 dB (full power). ARCHITECTURE's canonical order puts coarse power BEFORE field.edfs (tune_up.yaml step 1), and the step-order lint (protocol.py:191) covers only tune.auto_phase/tune.pi_calibration — exactly the cold-start case the overnight_t2 reorder fixed for phasing is still open for the vane loop.

**Scenario.** tune_up.yaml on a cold start with the magnet parked off the line: no echo anywhere, 4 length-nutation acquisitions of pure noise, up to 3 vane moves to essentially random attenuations (possibly 0 dB = maximum power into the resonator), then pi_length_target fails only if the last noise argmin happens to miss the +/-3.2 ns tolerance — and can spuriously PASS if it lands inside it, leaving the coarse stage 'done' at a garbage attenuation.

**Suggested fix.** Gate each iteration on the internal nutation's echo_snr (or phase_coherence) before computing a vane step, and record cal_judges into the log/manifest; extend the step-order lint to tune.power_for_length.

### 4. Dry-run pre-flights only the slowest rep_rate grid point; fastest rates unvalidated until live

`atomize/epr_auto/primitives/tune.py:444` — **CONFIRMED** · risk · dimension: state-flow

### 3. edfs target_snr projection stop can undershoot the hard echo_snr floor (inline claim overstated)

`atomize/epr_auto/primitives/field.py:123` — **PLAUSIBLE** · note · dimension: state-flow

**Defect.** field.py:121-126 claims 'the step registration floors target_snr at judges.SNR_FLOOR, so an early stop can never deliver a sweep the judge then rejects'. ### acquire_trace sends 'start' before the try/finally — a send failure orphans the worker

`atomize/epr_auto/engine/executor.py:256` — error-paths

In acquire_trace, `parent_conn.send('start')` at line 256 sits between process.start() (line 255) and the try block (line 260) whose finally is the only thing that join()s/terminate()s the child. run_worker does not have this gap (no pre-try send). Narrow window and unlikely trigger, hence a note, but the fix is trivial.

_Fix:_ Move parent_conn.send('start') inside the try block so the finally always runs, or guard it and treat a send failure as an EngineError after ensuring cleanup.

`atomize/epr_auto/primitives/exp.py:214` — error-paths

### P1. Inverse-length amp transfer ignores pulse envelope shape (GAUSS->SINE = 1.84x)

`atomize/epr_auto/primitives/tune.py:264` — **CONFIRMED** · bug · physics-numerics

**Defect.** `apply_calibration` (tune.py:256-264) and `_scale_detection_pair` (tune.py:699-700) transfer a fine amplitude calibration to a slot of a different length with `amp_slot = amp_cal * L_cal / L_slot`. That rule is only correct when the two pulses have the SAME envelope, because the flip angle is theta ∝ amp x L x <envelope>, not amp x L. The shape is available (`PulseSlot.typ`) but is only ever tested against BLANK/LASER (tune.py:205, :684); the envelope factor never enters either formula.

The Insys AWG applies the amplitude coefficient to the PEAK and multiplies by the envelope (Insys_FPGA.py:6556-6567: `y1 = norm_c*amp0/d_coef*sin(...)*envelope`, envelope peak 1; `d_coef = 100/amplitude`, Insys_FPGA.py:3007), so a GAUSS pulse of length L, sigma s carries only <envelope> = mean(exp(-0.5((n-L/2)/s)^2)) of a rect pulse's area. For the SHIPPED default amplitude-cal preset `ampl_4s.phase_awg` (P2 = GAUSS, length 28.8 ns, sigma 6.4 ns) I computed that factor exactly from the driver's own generator: 0.5434, i.e. 1/f = 1.840.

This is the default path, not an exotic one: `_check_pi_calibration` (steps.py:161-164) defaults `mode: amplitude` to `ampl_4s.phase_awg`, whose swept pulse is the GAUSS; every shipped experiment preset the calibration is then applied to (`hahn_echo_4s`, `ed_4s`, `inversion_recovery_echo_4s_log`) uses SINE pulses. ARCHITECTURE.md:131-140 states the rule and even names the shape mismatch ('sweeps a 28.8 ns GAUSS and detects with an 86.4 ns SINE pair') while prescribing the pure length ratio, so the gap is not blessed anywhere — the justification given is amplifier linearity, which is a different (and correct) point.

**Scenario.** Run `protocols/overnight_t2.yaml` (tune.pi_calibration mode: amplitude on ampl_4s, then exp.t2 with inferred apply_cal on hahn_echo_4s). I reproduced the transfer in `run --test`: `apply_cal -> P2: amplitude=43.07 % (pi2), P3: amplitude=43.71 % (pi)`. With the GAUSS shape factor 0.5434 the physically correct values are 23.41 % and 23.75 %, so the pulses are driven 1.84x too hard: the 'pi/2' is a 166 deg rotation and the 'pi' a 331 deg rotation. Hahn echo amplitude sin(t1)*sin(t2/2)^2 drops from 1.000 to 0.0153 — a ~65x signal loss. exp.t1 is identical (`apply_cal -> P2: 61.2 % (pi), P3: 30.15 % (pi2), P4: 61.2 % (pi)` on the inversion-recovery preset). Nothing catches it: the only guard on the scaled value is `> 100 %`.

**Suggested fix.** Carry an envelope area factor with the calibration and divide/multiply by the ratio: theta ∝ amp * L * f(typ, sigma, L), with f = 1 for SINE, mean(exp(-0.5((n-L/2)/sigma)^2)) for GAUSS, the corresponding sinc/WURST/SECH integrals otherwise. Store `shape_factor` (or the raw typ/sigma/length) in the pi_calibration result next to `length_ns`, and use `amp_slot = amp_cal * (L_cal*f_cal)/(L_slot*f_slot)` in both apply_calibration and _scale_detection_pair. As a minimum stop-gap, refuse to transfer (or loudly warn) when `cal_slot.typ != target_slot.typ`.

### echo_window smooths with a box whose width scales with the trace, not the echo (FWHM +76%)

`atomize/epr_auto/primitives/tune.py:320`

`echo_window` measures the echo FWHM on `_smooth(np.abs(sig))` (tune.py:320) and sets the integration window to FWHM*factor. `_smooth` (tune.py:554-556) uses a boxcar of `max(3, len(y)//20)` points — a width tied to the LENGTH OF THE ACQUISITION TRACE, not to the echo. At decimation 1 a 512 ns detection window is 1280 samples, so the box is 65 points = 26 ns, comparable to a real echo width, and it broadens the measured FWHM. Measured through the shipped `_smooth` plus echo_window's own half-maximum index logic on synthetic Gaussian echoes (tpp = 0.4 ns):
  512 ns window, true FWHM 60 ns -> measured 62.4 ns (+4 %)
  512 ns window, true FWHM 30 ns -> measured 35.2 ns (+17 %)

### adjusted-R2 degrees of freedom off by one for the way n_params is counted

`atomize/epr_auto/primitives/judges.py:114`
